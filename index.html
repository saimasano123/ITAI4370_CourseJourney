My Learning Journey

When I began this course, telecommunications felt like a black box. I knew how to use my phone, Wi-Fi, and the internet, but I had very little understanding of what actually happened between a transmitter and a receiver. Throughout this course, every lecture, assignment, lab, and project added another layer to that understanding. Eventually, I realized that modern communication networks are much more than systems that simply move data—they are intelligent systems that continuously make decisions using artificial intelligence. This website documents that journey.

Midterm Project

Three progressively advanced network simulations, combining discrete-event simulation, agent-based modeling, and machine learning:

1. SimPy Network Simulation — A three-router packet-processing chain modeled as a discrete-event simulation. Result: average latency ≈ 8.10 time units, throughput ≈ 0.370 packets/time unit across 10 packets.

2. Mesa Agent-Based Network — Five routers on a Watts–Strogatz network graph, each with an independent random-walk load. Router 5 crossed the 80% congestion threshold for four consecutive steps (35–38) and triggered rerouting messages before recovering.

3. Machine Learning Traffic Prediction — A scikit-learn Linear Regression model predicting next-step traffic from the previous time step (lag-1 feature) on synthetic sine-wave-plus-noise data. Result: MSE ≈ 18.449.

Reflection: This project made me realize my understanding is stronger on the "what does this code do" level than the "why this technique over another" level — especially in the agent-based model, where I could follow the mechanics but wasn't fully confident why an agent-based approach was preferable to the resource-allocation math from Module 4. That's a gap I want to close going forward.

Final Reflection

Essay 1 — How My Understanding of Telecommunications Evolved

I started this course thinking of telecommunications as "having good signal on my phone" — a black box I used daily without any real model of it. Module 1's transmitter–channel–receiver model broke that open: the BPSK simulation in Lab 1 showed a signal buried in noise still being recovered correctly, which made "noise resilience" a visible, measurable property instead of a marketing phrase.

Each module then added a layer of architecture on top. Module 2's FSPL lab explained why cell towers are spaced the way they are; Module 4's slice-allocation exercise made network slicing concrete in code, not just analogy. By Module 5, my mental model had shifted from a static pipe carrying data to a dynamic system constantly trading off latency, bandwidth, and reliability.

The biggest shift was realizing telecommunications isn't just infrastructure — it's infrastructure with decisions embedded in it, and increasingly, AI is what's making those decisions.

Essay 2 — Applying AI to Telecommunications Problems

Lab 3's Random Forest traffic model taught me that good feature engineering can matter as much as the model itself — the model discovered daily/weekly patterns directly from lag features, without being told the schedule. That lesson kept resurfacing all course.

Lab 5 turned "AI for the edge" into a real engineering tradeoff: the quantized model kept 93.4% accuracy while shrinking from 51 KB to 22 KB, while the distilled model shrank further but lost more accuracy. Lab 4 then let me compare ARIMA, Linear Regression, and an LSTM on the same forecasting problem — watching ARIMA flatten toward the mean while the LSTM tracked the daily cycle almost automatically (R² ≈ 0.89) taught me more about statistical vs. deep-learning approaches than any lecture could.

If I'm honest, my remaining gap is less about running these models and more about confidently choosing between approaches — seen clearly in the Midterm's agent-based router simulation, where I could follow the mechanics but wasn't fully sure why an agent-based model was the right tool over the resource-allocation math from Module 4. That's the skill I most want to build next: not just implementing a technique, but justifying why that technique, for this problem.
